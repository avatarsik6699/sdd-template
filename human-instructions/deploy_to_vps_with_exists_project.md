Вот порядок действий:

### 1. Остановить и удалить старое приложение

```bash
# Перейти в папку старого проекта
cd /opt/старый-проект   # или где он лежит

# Остановить контейнеры и удалить volumes старой БД
docker compose down -v

# Удалить папку проекта
cd /
rm -rf /opt/старый-проект
```

### 2. Почистить Docker полностью

```bash
# Удалить все остановленные контейнеры, неиспользуемые образы, сети
docker system prune -af
docker system prune -a --volumes

# Проверить, что ничего не осталось
docker ps -a       # пусто
docker images      # пусто
docker volume ls   # пусто (если volumes удалили на шаге выше)
```

### 3. Проверить порты

```bash
ss -tlnp | grep -E '80|443|8000|3000|5432|6379'
# Всё должно быть свободно
```

### 4. Дальше по DEPLOY.md начиная с шага 5

SSL-сертификат и домен уже есть — пропустить шаги 3 (DNS) и 6 (Certbot), если домен тот же.

Если **домен тот же**, сертификат в `/etc/letsencrypt/` уже лежит — просто замените `[DOMAIN]` в `nginx.conf` и запускайте стек. Certbot-сервис подхватит существующие сертификаты и будет их продлевать автоматически.

Если **домен другой** — нужно выпустить новый сертификат (шаг 6).

---

> **Осторожно с `docker system prune -af`:** удаляет все образы на хосте, включая те, что могут использоваться другими проектами. Если на сервере есть что-то ещё — используйте `docker compose down -v` только в папке нужного проекта, без глобальной очистки.

Nginx запущен **прямо на хосте** (не в Docker) — это системный процесс, не контейнер. Нужно его остановить и отключить автозапуск:

```bash
sudo systemctl stop nginx
sudo systemctl disable nginx   # чтобы не поднялся после перезагрузки
```

Проверить:

```bash
ss -tlnp | grep -E '80|443'
# Должно быть пусто
```

После этого порты 80 и 443 свободны — можно запускать стек по DEPLOY.md (шаг 6, Certbot, затем шаг 7).
