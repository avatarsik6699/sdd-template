## **🛡️ Что такое фаервол (firewall)?**

Представьте, что ваш VPS — это дом:
- **Порты** — это двери и окна дома
- **Фаервол** — это система замков и решеток
- **UFW** — простой пульт управления этой системой

**Без фаервола:** Все 65535 "дверей" (портов) открыты. Любой может попробовать зайти в любую "дверь".

**С фаерволом:** Открыты только те "двери", которые нужны для работы.

---

## **📡 Что такое порты?**

**Порт** — это виртуальный "почтовый ящик" для сетевых соединений. У каждого сервиса свой ящик:

| Порт | Сервис | Аналог в доме | Для чего |
|------|--------|---------------|----------|
| 22 | SSH | Секретный вход для хозяина | Управление сервером |
| 80 | HTTP | Парадная дверь для гостей | Веб-сайт без шифрования |
| 443 | HTTPS | Бронированная дверь | Веб-сайт с шифрованием |
| 21 | FTP | Складская дверь | Передача файлов (часто взламывают!) |
| 3306 | MySQL | Сейф с данными | База данных (опасно оставлять открытой!) |

**Всего портов:** 0-65535
- **0-1023:** Системные (привилегированные)
- **1024-49151:** Зарегистрированные (популярные сервисы)
- **49152-65535:** Динамические (временные соединения)

## **🛠️ UFW (Uncomplicated Firewall) — Простой фаервол**

### **Почему именно UFW?**
Раньше в Linux использовался `iptables` — мощный, но сложный:
```bash
# Старый способ (iptables)
sudo iptables -A INPUT -p tcp --dport 22 -j ACCEPT
sudo iptables -A INPUT -p tcp --dport 80 -j ACCEPT
sudo iptables -A INPUT -p tcp --dport 443 -j ACCEPT
sudo iptables -A INPUT -j DROP
```

**UFW упрощает это до:**
```bash
sudo ufw allow 22/tcp
sudo ufw allow 80/tcp
sudo ufw allow 443/tcp
sudo ufw enable
```

## **🔍 Детальный разбор каждой команды**

### **1. `sudo ufw allow 22/tcp`**

**Что делает:** Разрешает входящие соединения на порт 22 по протоколу TCP.

**Почему TCP, а не UDP?**
SSH использует TCP потому что:
- **TCP** гарантирует доставку пакетов в правильном порядке (как заказное письмо)
- **UDP** отправляет "вслепую" без подтверждения (как открытка)

**Что если не открыть порт 22?**
```bash
$ ssh user@ваш-сервер
ssh: connect to host ваш-сервер port 22: Connection refused
```
**Вы потеряете доступ к серверу!** Придется обращаться в поддержку VPS.

**Безопасность порта 22:**
```bash
# Рекомендуемые дополнительные меры:
sudo ufw limit 22/tcp  # Ограничивает частоту подключений
# ИЛИ
sudo ufw allow from 192.168.1.100 to any port 22  # Только с вашего IP
```

### **2. `sudo ufw allow 80/tcp`**

**Зачем открывать порт 80, если у нас HTTPS?**
1. **Редирект:** Пользователь вводит `http://site.com` → автоматически переходит на `https://site.com`
2. **Проверка Let's Encrypt:** Certbot использует порт 80 для проверки владения доменом

**Что видит пользователь:**
```
Браузер: http://ваш-сайт
Сервер: 301 Moved Permanently → https://ваш-сайт
```

### **3. `sudo ufw allow 443/tcp`**

**HTTPS = HTTP + TLS/SSL**
- Порт 443 обрабатывает зашифрованные соединения
- Все современные браузеры требуют HTTPS

**Технически:**
```
Клиент: "Client Hello" (запрос на порт 443)
Сервер: Отправляет SSL сертификат
Клиент: Проверяет сертификат
Оба: Создают общий ключ шифрования
Начинается зашифрованная передача данных
```

### **4. `sudo ufw enable`**

**Самая опасная команда!** Почему?

**До `enable`:**
```bash
$ sudo ufw status
Status: inactive
# Правила записаны, но не применяются
```

**После `enable`:**
```bash
$ sudo ufw status
Status: active
# Все правила вступили в силу
# ВСЕ остальные порты ЗАКРЫТЫ
```

**Что закрывается:**
- Порты 21 (FTP) — часто взламывают
- Порты 25 (SMTP) — спамеры ищут открытые
- Порты 3306 (MySQL), 5432 (PostgreSQL) — базы данных
- Порты 27017 (MongoDB) — часто мишень для хакеров
- Все остальные 65532 порта!

## **🎯 Что происходит внутри UFW?**

### **Таблицы правил iptables:**
После выполнения команд UFW создает такие правила:
```
Chain INPUT (policy DROP)
target     prot opt source     destination
ACCEPT     tcp  --  anywhere   anywhere    tcp dpt:ssh
ACCEPT     tcp  --  anywhere   anywhere    tcp dpt:http
ACCEPT     tcp  --  anywhere   anywhere    tcp dpt:https
ACCEPT     all  --  anywhere   anywhere    state RELATED,ESTABLISHED

Chain FORWARD (policy DROP)
target     prot opt source     destination

Chain OUTPUT (policy ACCEPT)
target     prot opt source     destination
```

**Ключевое правило:**
```bash
ACCEPT all -- anywhere anywhere state RELATED,ESTABLISHED
```
Разрешает ответные пакеты на ваши исходящие запросы.

## **🚨 Типичные ошибки новичков**

### **Ошибка 1: Заблокировать себя (SSH)**
```bash
# НЕПРАВИЛЬНО:
sudo ufw enable  # Включаем до добавления правил
# Результат: доступ по SSH потерян!
```

**Правильная последовательность:**
```bash
1. sudo ufw allow 22/tcp   # Сначала разрешить SSH
2. sudo ufw enable         # Потом включить
```

### **Ошибка 2: Открыть все порты**
```bash
# НЕПРАВИЛЬНО:
sudo ufw default allow  # Разрешить всё
# Теперь ваш сервер как дом без стен
```

### **Ошибка 3: Забыть проверить статус**
```bash
# Всегда проверяйте:
sudo ufw status verbose
# Или с номерами правил:
sudo ufw status numbered
```

## **🔧 Полезные команды UFW**

### **Просмотр статуса:**
```bash
# Подробный статус
sudo ufw status verbose

# С номерами правил (удобно для удаления)
sudo ufw status numbered

# Показать добавленные правила
sudo ufw show added
```

### **Управление правилами:**
```bash
# Разрешить порт с указанием протокола
sudo ufw allow 53/udp  # DNS
sudo ufw allow 123/udp # NTP (время)

# Разрешить диапазон портов
sudo ufw allow 8000:8010/tcp  # Для тестирования

# Разрешить по имени службы
sudo ufw allow ssh
sudo ufw allow http
sudo ufw allow https

# Удалить правило
sudo ufw delete allow 80/tcp
# Или по номеру из status numbered
sudo ufw delete 2

# Сбросить все правила
sudo ufw reset  # ОСТОРОЖНО! Закроет всё
```

### **Ограничение вместо разрешения:**
```bash
# limit - разрешает, но блокирует при частых подключениях
sudo ufw limit 22/tcp  # Защита от брутфорса

# reject - вежливый отказ
sudo ufw reject 25/tcp  # Отправляет "порт закрыт"

# deny - тихий отказ
sudo ufw deny 21/tcp    # Игнорирует запросы
```

## **🌐 Пример атаки без фаервола**

### **Сканирование портов хакером:**
```bash
# Хакер запускает:
nmap ваш-сервер

# Результат без фаервола:
PORT     STATE SERVICE
21/tcp   open  ftp      # Уязвимости!
22/tcp   open  ssh      # Брутфорс паролей
25/tcp   open  smtp     # Спам-реле
3306/tcp open  mysql    # База данных открыта!
5432/tcp open  postgresql # Ваши данные!

# Хакер пробует стандартные пароли:
hydra -l root -P passwords.txt ssh://ваш-сервер
```

### **С фаерволом:**
```bash
nmap ваш-сервер

PORT    STATE SERVICE
22/tcp  open  ssh     # Только SSH
80/tcp  open  http    # Только веб
443/tcp open  https   # Только HTTPS
```

## **🛡️ Дополнительные меры безопасности**

### **1. Изменение порта SSH (необязательно, но полезно):**
```bash
# В файле /etc/ssh/sshd_config:
Port 22222  # Меняем с 22 на случайный

# В UFW:
sudo ufw allow 22222/tcp
sudo ufw deny 22/tcp
```

### **2. Fail2ban — защита от перебора паролей:**
```bash
sudo apt install fail2ban
# Автоматически блокирует IP после 5 неудачных попыток SSH
```

### **3. Блокировка по странам (geoblocking):**
```bash
# Установка ipset
sudo apt install ipset

# Блокировка Китая, России и т.д.
sudo ufw deny from 1.0.0.0/8     # Китай
sudo ufw deny from 14.0.0.0/8    # Китай
# и т.д.
```

## **📊 Проверка безопасности**

### **Проверка открытых портов извне:**
```bash
# С другого сервера или через онлайн-сервис:
nmap -Pn ваш-сервер

# Должно показать только:
22/tcp, 80/tcp, 443/tcp
```

### **Проверка на утечки DNS:**
```bash
# Установить и запустить:
sudo apt install dnsutils
dig ваш-домен
# Проверить, не "протекает" ли информация о сервере
```

## **🚨 Чрезвычайные ситуации**

### **Если заблокировали себя по SSH:**
**Решение 1:** Консоль VPS провайдера (DigitalOcean, Hetzner имеют веб-консоль)

**Решение 2:** Если есть физический доступ

**Решение 3:** Обратиться в поддержку

### **Временное отключение фаервола:**
```bash
sudo ufw disable  # Отключает фаервол
# НЕ ДЕЛАТЬ на работающем сервере!
```

## **🎯 Ваша конкретная ситуация**

В вашем деплое фаервол настраивается **в начале**, потому что:

1. **Безопасность прежде всего** — пока вы настраиваете сервер, он уязвим
2. **SSH должен быть открыт сразу** — иначе потеряете доступ
3. **80 и 443 для веб-сервера** — Nginx будет слушать эти порты

**Что будет, если пропустить этот шаг?**
- Ваш PostgreSQL (5432) может быть доступен из интернета
- Redis (6379), если будете использовать, тоже откроется
- Боты начнут сканировать порты через минуту после запуска сервера

## **📝 Шпаргалка по безопасности портов**

### **Всегда закрывать:**
```
21    - FTP (устарел, небезопасен)
23    - Telnet (пароли в открытом виде!)
25    - SMTP (если не почтовый сервер)
135-139, 445 - Windows shares (для Linux не нужны)
3306  - MySQL (только локально)
5432  - PostgreSQL (только локально)
3389  - RDP (для Windows)
```

### **Открывать с осторожностью:**
```
22    - SSH (лучше с ключами, без паролей)
8080  - Dev сервер (только для разработки)
3000  - Node.js dev (только локально)
```

### **Безопасно открывать:**
```
80    - HTTP (для редиректа на HTTPS)
443   - HTTPS (основной порт)
```
